# MiniKVX-Agent 评测计划

评测目标不是做复杂性能平台，而是让项目从“能跑”变成“能证明为什么这样设计”。所有数字必须真实测出，不能为了简历虚构。

## 评测 1：Redis 锁防重复执行

### 目标

验证同一个任务被并发触发时，只有一个请求真正进入执行流程，避免重复执行和日志污染。

### 场景

- 准备一个可执行任务 `taskId`。
- 并发发送 10 个 `POST /api/tasks/{taskId}/run` 请求。
- 预期 1 个请求获得执行锁，其余请求返回“任务正在执行”。

### 记录指标

- 并发请求数。
- 成功获得执行锁的请求数。
- 被 Redis 锁拦截的请求数。
- 任务日志中实际执行次数。

### README 表达

```text
并发触发同一任务 10 次时，仅 1 次获得执行锁，其余请求被拦截，避免重复执行和日志污染。
```

## 评测 2：同步执行 vs RabbitMQ 异步投递

### 目标

证明 RabbitMQ 不是硬塞技术栈，而是用于解耦任务提交和任务执行，让 HTTP 接口快速返回。

### 场景

对比两种模式：

1. 同步执行：`POST /api/tasks/{taskId}/run` 等任务执行完成再返回。
2. 异步投递：接口只投递 taskId 到 RabbitMQ 后返回，Worker 后台消费执行。

### 记录指标

- 请求样本数量。
- 平均响应耗时。
- P95 响应耗时。
- 最小/最大耗时。
- 任务最终成功数。

### README / 简历表达

```text
引入 RabbitMQ 解耦任务提交与执行流程，并通过评测对比同步执行和异步投递两种模式下的接口耗时，验证任务提交接口从秒级等待优化为毫秒级返回。
```

## 评测 3：WorkerPool 并发度对比

### 目标

评估不同 worker 数量对任务执行吞吐的影响，证明 WorkerPool 不是只写 goroutine，而是有并发度控制和调优意识。

### 场景

固定任务数量和步骤耗时，分别测试：

```text
worker=1
worker=2
worker=4
worker=8
```

### 记录指标

- 总任务数。
- worker 数量。
- 总耗时。
- 每秒完成任务数。
- 平均任务等待时间，可选。
- 错误数。

### 面试准备

- worker 越多一定越好吗？
- 什么时候瓶颈会从 CPU/执行逻辑转移到 MySQL 或日志写入？
- 数据库连接池和 WorkerPool 并发度有什么关系？

## 工具建议

- PowerShell / curl：适合简单接口验证。

## 评测 4：RAG 检索

### 目标

验证 RAG 能检索到正确来源，而不是只展示“能调用大模型”。

### 场景与指标

- 准备 10-20 条问题和预期来源文档。
- 统计 `Hit@3`：Top3 结果是否包含预期来源。
- 记录平均检索耗时和 P95。
- 检查最终回答是否返回来源引用。

### 限制

- Hit@3 只验证检索命中，不等于回答完全正确。
- v2 暂不引入复杂 LLM-as-a-Judge 或人工大规模标注。
